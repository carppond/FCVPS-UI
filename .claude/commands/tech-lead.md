## 检查项

1. UI 设计的交互是否能被架构方案支撑？（如 UI 要实时更新，架构是否有 WebSocket/SSE）
2. UI 的组件/页面和架构的模块是否对应？
3. 架构的接口设计是否覆盖了 UI 需要的所有数据？
4. 有没有需求中的功能被两边都遗漏了？
5. 技术选型是否合理？是否有更简单的替代方案？
6. API 契约是否完整覆盖了所有接口？类型定义是否和架构文档一致？

## 输出要求

用 Write 工具创建 `docs/05-tech-lead-plan.md`，内容包含：

### 1. 冲突与修正
UI 和架构之间的冲突及裁决。无冲突写"无冲突"。

### 2. 遗漏补充
被遗漏的功能，补充到哪个模块。

### 3. 开发任务分解
拆分为具体任务，每个包含：
- 任务名称
- 文件路径列表（精确到文件）
- 依赖的其他任务
- 实现说明

原则：
- "任务 0：项目初始化与共享文件"包含所有共享文件（package.json / tsconfig / 入口文件 / 通用配置等）
- API 契约生成的类型定义文件属于共享文件、已存在，**任务 0 不重新创建**，只确认路径正确
- 其余按模块拆分，文件不重叠
- 标注依赖关系

### 4. 技术风险
可能踩的坑。

## 规则
- 你是最终裁决者
- 每个文件只能属于一个任务
- 任务完整覆盖架构文档中的所有文件
