# 编码规范

## 1. 文件粒度

- 单文件上限 **300 行**（软限制）：超过时必须审视是否可以按职责拆分
- 单函数/方法上限 **50 行**：超过意味着做了太多事，需要提取子函数
- 单组件上限 **200 行**（前端）：超过则拆分子组件
- 拆分原则：按**职责**拆，不是按行数凑。一个文件只做一件事

## 2. 注释要求

- **文件头注释**：仅在文件名不能完全表达用途时添加，一行即可。`user-service.ts` 不需要，`transformer.ts` 需要说明转换什么
- **公共函数/方法**：必须添加注释，说明参数含义、返回值、副作用
- **内部函数**：仅在"为什么这样做"不明显时添加
- **行内注释**：解释 why，不解释 what
- **TODO**：允许，格式为 `// TODO(原因): 具体内容`，禁止无原因的裸 TODO

## 3. 代码复用

- **重复阈值**：同样逻辑出现 **3 次**时提取为共享函数/组件。2 次可以容忍，3 次必须抽取
- **工具函数**：统一放 `utils/` 或 `helpers/` 目录
- **共享类型**：统一放 `types/` 或 `shared/` 目录
- **共享组件**：统一放 `components/common/` 或 `components/shared/`
- 提取时确保新函数/组件的命名能脱离原上下文独立理解

## 4. 命名规范

总原则：**跟随语言社区惯例**，以下为默认规则，具体语言有冲突时以语言惯例为准。

- **文件名**：kebab-case（JS/TS）、snake_case（Go/Python/Rust）、PascalCase（Java/C#）
- **变量和函数**：camelCase（JS/TS/Java）、snake_case（Go/Python/Rust）
- **类/类型/接口**：PascalCase
- **常量**：UPPER_SNAKE_CASE
- **布尔变量**：使用 `is/has/should/can` 前缀，如 `isLoading`、`hasPermission`
- **事件处理函数**：使用 `handle` 前缀或 `on` 前缀，如 `handleSubmit`、`onClick`
- 命名要有语义，禁止 `data`、`info`、`temp`、`result` 等无意义名称（循环变量 `i/j/k` 除外）

## 5. 性能要求

- **循环嵌套**：不超过 3 层，超过需要重构（提取函数、用 Map 替代查找等）
- **异步操作**：无依赖关系的异步操作必须并行执行（Promise.all / goroutine / asyncio.gather）
- **前端渲染**：列表必须有稳定的 key；超过 100 项的列表考虑虚拟滚动；避免在渲染路径中做重计算
- **数据库查询**：禁止 N+1 查询，必须批量获取；查询条件字段必须有索引
- **缓存**：频繁读取且不常变化的数据应考虑缓存
- **包体积（前端）**：使用动态 import 做代码分割，避免全量导入大型库（如 `import { specific } from 'lodash/specific'`）

## 6. 错误处理

- **外部调用**：所有 API 调用、数据库操作、文件操作必须有错误处理
- **错误信息**：包含上下文信息（什么操作、什么参数、什么期望），方便排查
- **用户可见错误**：友好提示，不暴露内部堆栈或数据库结构
- **兜底处理**：前端全局错误边界、后端全局异常中间件
- **不吞异常**：catch 后必须处理（记录日志 / 向上抛出 / 返回错误），禁止空 catch

## 7. 项目结构

### 后端
```
src/
├── routes/        # 路由定义
├── controllers/   # 请求处理（参数校验、调用 service、返回响应）
├── services/      # 业务逻辑
├── models/        # 数据模型
├── middleware/     # 中间件
├── utils/         # 工具函数
├── types/         # 类型定义
└── config/        # 配置
```

### 前端
```
src/
├── pages/         # 页面级组件
├── components/    # 可复用组件
│   ├── common/    # 通用组件（Button、Modal 等）
│   └── feature/   # 业务组件
├── hooks/         # 自定义 hooks
├── services/      # API 调用
├── stores/        # 状态管理
├── utils/         # 工具函数
├── types/         # 类型定义
└── assets/        # 静态资源
```

### 通用原则
- 路由/控制器 → 服务层 → 数据层，严格分层，禁止跨层调用
- 配置统一管理，不散落在代码中
- 测试文件和源码同目录或统一 `__tests__/` 目录
