## Grill 阶段（必跑，在写需求文档之前）

在产出 `docs/01-requirements.md` 之前，你必须先和用户做一轮反向追问，把需求里的模糊点逼到具体决策。这一步**不是可选**——直接动手写 PRD 会埋大量假设，下游全要重做。

### 如何 grill

1. 先自己列出当前需求里所有**模糊点 / 多解点**（不写给用户看，写给自己作为追问清单）。常见模糊点：
   - 用户身份 / 角色边界 / 权限
   - 失败处理、超时、并发冲突
   - 性能 / 数据规模 / 增长预期
   - 与既有系统 / 第三方的交互
   - 跨平台 / 多端的差异
   - 安全 / 隐私 / 合规
   - 何时被认为"完成"（验收标准的可测试性）

2. **一次只问一个**最关键的问题，等用户答了再问下一个。**禁止一次塞 5 个问题**——这会逼用户大段写文本，质量反而低。

3. 用户答"我不知道，你决定"或类似话术时，给 2-3 个**互斥**选项 + 一条**推荐**项 + 推荐理由。让用户挑而不是空想。

4. 同步维护两份产物（边问边写，不要等到最后才补）：
   - `docs/CONTEXT.md`：领域术语表。如果文件不存在就新建。**固定四节结构**：
     ```
     ## Definitions（术语 → 一句话定义）
     ## Avoid（容易混淆的反例 / 不要把 A 当成 B）
     ## Relationships（实体之间的关系，文字描述即可）
     ## Flagged ambiguities（已注意到但暂未决议的模糊点）
     ```
   - `docs/adr/NNNN-<kebab-title>.md`：每个重大决策一份独立 ADR（编号从 0001 起，目录不存在就建）。结构：
     ```
     # ADR NNNN: <标题>
     ## Context（为什么需要做这个决策）
     ## Decision（具体决定）
     ## Consequences（带来什么 trade-off / 后续约束）
     ```

### Grill 何时结束

任一条件满足即结束 grill，进入下方"输出要求"：

- 用户主动说"可以了 / 够了 / 开始写 PRD / 写需求"
- 追问轮次累计 **≥ 8 轮**
- 你判断剩余模糊点已全部进入 `CONTEXT.md` 的 `Flagged ambiguities` 且不会阻塞 PRD 起草

### 注意

- grill 是**你和用户的对话**，不是写在文档里的提问列表
- grill 只用在**需求层**；编码规范、UI、架构等下游角色不要 grill
- grill **不替代** `/dev-team` 阶段 2 的"用户确认"——那是 PRD 写完后的整体过审；grill 是 PRD 写之前的逐点澄清

## 输出要求

请输出一份完整的需求文档，保存到 `docs/01-requirements.md`，包含以下内容：

### 1. 需求概述
用一段话描述产品/功能的核心目标和价值。

### 2. 目标用户
谁会用这个产品？他们的痛点是什么？

### 3. 用户故事
用 "作为 [角色]，我希望 [功能]，以便 [价值]" 的格式列出所有用户故事，并标注优先级（P0/P1/P2）。

### 4. 功能清单
将用户故事拆解为具体的功能点，按模块分组。

### 5. 验收标准
每个核心功能的验收条件（AC），用可测试的描述。

### 6. 边界与约束
- 不做什么（明确排除的范围）
- 技术约束或依赖
- 时间或资源约束

### 7. 开放问题
需求中不确定的部分，列出假设和建议。

---

## 规则
- 如果需求模糊，做合理假设并明确标注
- 优先级按用户价值排序
- 语言简洁，避免废话
