## 审查维度

### 1. 编码规范
对照 `docs/00-coding-standards.md` 逐项检查：单文件 ≤300 行、单函数 ≤50 行、单组件 ≤200 行（前端）、注释、命名、复用阈值（3 次必抽取）、错误处理、禁止空 catch。

### 2. 功能正确性
- 是否实现了需求中的全部功能
- 边界情况
- 明显逻辑漏洞

### 3. 代码质量
- 命名清晰、职责单一
- 重复代码（出现 3 次以上未提取的）
- 复杂度合理

### 4. 安全性
- OWASP Top 10
- 输入验证 / 输出编码
- 敏感信息处理（不入库、不打印日志）
- 依赖已知漏洞

### 5. 性能
- 循环嵌套不超过 3 层
- 无依赖异步操作必须并行
- N+1 查询、未加索引的查询字段
- 前端：列表 key、不必要重渲染、包体积

### 6. API 契约一致性
对照 `docs/04-api-contract.md` 检查接口调用的请求/响应格式、错误码、字段名、可选/必填是否一致。前后端类型必须 import 契约类型，禁止自定义重复类型。

### 7. 依赖安全
按平台运行扫描，记录高危/严重漏洞：
- Node 项目：`npm audit --audit-level=high`
- Python：`pip-audit` 或 `safety check`
- Go：`govulncheck ./...`
- Rust：`cargo audit`

### 8. 视觉规范（仅前端项目）
对照 `docs/02-ui-design.md` 的 Design Tokens 和 Anti-patterns：
- 用 Grep 搜源码（**排除 token 文件本身**：`tailwind.config.*`、`src/styles/tokens.*`、`src/theme.*`）：
  - `#[0-9a-fA-F]{3,8}\b` 命中数应为 0（裸 hex 是违规）
  - `\brgb\(|\brgba\(|\bhsl\(|\bhsla\(` 命中数应为 0（裸颜色函数是违规）
- **任意像素判定**：用 Grep 扫 `\b\d+px\b`，逐个命中对照 `docs/02-ui-design.md` 中的间距 / 字号 / 圆角 / 阴影 / 动效阶梯。**不在阶梯内的像素值即违规**。豁免白名单：
  - `0px`
  - `1px` 用于边框（如 `02-ui-design.md` 风格关键词包含 `hairline`）
  - token 文件本身（已排除）
- Anti-patterns 一条不能违反（逐条核对 02-ui-design.md 里列出的）

## 问题分类（输出严重问题时必填）

每条"严重问题" / "建议改进"必须以 `[bug]` 或 `[style]` 开头标注类型：

- `[bug]` — 功能错误 / 逻辑漏洞 / 数据损坏 / 安全漏洞 / 性能错误（死循环、N+1 等）。**下游修复时必须走 Diagnose 六步法**（复现 → 最小化 → 假设 → 插桩 → 修复 → 回归），不能凭直觉直接改。
- `[style]` — 命名 / 格式 / lint / 类型注解 / 注释缺失 / 文件超长 / 重复代码。直接改即可，不走六步法。

判断口诀：**"用户能感知或会出错就是 bug；只是看着不舒服就是 style"**。

## Diagnose 六步法（供下游修复 Agent 用，本节供 reviewer 知晓）

任何标 `[bug]` 的问题，修复 Agent 都要按下面六步走：

1. **复现**：先写最小复现脚本 / 失败测试，确认稳定触发；复现不出就停下来报告，不要硬猜。
2. **最小化**：删除复现脚本里所有与 bug 无关的代码，留最小骨架。
3. **假设**：列 2-3 个**互斥**可能原因，禁止"A 也可能 B 也可能 C"一锅炖。
4. **插桩**：在每个假设的关键路径加 log / 断点（**不要先改代码**），跑最小复现看哪个假设成立。
5. **修复**：只修验证成立的那个根因。不顺手改别的、不顺手重构。
6. **回归**：第 1 步的复现脚本变成正式回归测试塞进测试套件，确认通过。

第 2 轮修复的特例：第 1-2 步可跳过（round1 已做过），但 3-6 必须重走，不能套用 round1 的假设。

review 时不需要你自己跑六步——你只负责**正确分类** `[bug]` / `[style]`，让下游修复 Agent 知道该走哪条路径。
