### 1.2 错误码 `ERR_AUTH_TOTP_REQUIRED` HTTP 状态 202（轻微冲突）

**冲突**：契约 §3 把 `ERR_AUTH_TOTP_REQUIRED` 标为 HTTP 202 Accepted（"密码通过但需 2FA"）。但 §4.1 登录流程描述的是"未启用 2FA 返 access_token；已启用 2FA 返 pending_token"，是两类不同响应而不是错误。

**裁决**：**改语义为"成功响应的两种 variant"，不返回 error code**。
- 登录密码正确 → 200 OK；data 是 `LoginResponse`（含 access_token）或 `PendingTOTPResponse`（含 pending_token + `totp_required: true`）。
- 前端通过 `data.totp_required === true` 判断分支。
- `ERR_AUTH_TOTP_REQUIRED` 错误码保留但仅用于"在需要 totp 的中间状态下访问受保护接口"场景。
- 契约文档不需要改字段，但 §4.1 描述需要明确"密码 + 2FA 的两种 variant 是正常 200 响应而不是 202"。

**需修订**：**需修订 docs/04-api-contract.md §3** 把 `ERR_AUTH_TOTP_REQUIRED` 的"HTTP 状态"由 `202` 改为 `200`（语义改为"中间状态保护"），并在 §4.1 补一句"两种 variant 都是 200，前端通过 totp_required 字段判断"。标记交后续 Agent 处理。

### 1.3 sub-store 兼容路由的"public+token"在静默模式下行为（澄清）

**冲突**：架构 §5.3 静默模式允许列表包含 `/download/:name`（必带 token），但前端文档（UI 设计 §流程 5）未明说短链 / sub-store 兼容路径的 token 来源。契约 §6 列出 `GET /download/:name?token=<token>` 但未说明 token 的获取入口。

**裁决**：
- **token 来源**：订阅详情页（`/subscriptions/:id`）展示 sub-store 兼容 URL，包含订阅级 token（由后端在订阅创建时一并生成，与登录 session token 不同；存在 `subscriptions` 表的新字段 `share_token TEXT NOT NULL`）。
- **生成方式**：32 字节 base64url，per-subscription，不可被 admin/user 在 UI 直接看到 hash 后的值（与 agent token 行为一致），但可在订阅详情页"复制兼容 URL"按钮一键复制完整 URL。
- **轮换**：订阅详情页提供"轮换 token"按钮，吊销旧 URL。

**需修订**：

- **需修订 docs/04-api-contract.md §2.4 Subscription DTO** 与 `internal/types/api.go` 的 `Subscription` struct：新增 `share_token` 字段（仅在订阅详情接口返回，列表不返回；string，必填）。
- **需修订 docs/04-api-contract.md §1** 端点表：新增 `POST /api/subscriptions/:id/rotate-share-token`（合计 endpoint 数从 102 升到 103）。
- 任务 T-11 落地此字段；其他任务 T-12（substore 兼容 handler）依赖此字段。

### 1.4 流水线编辑器"前端运行 + 后端预览"两种模式（UI 设计第 6 组件遗漏的细节）

**UI 文档**（02-ui-design.md §流程 3）描述"GUI 拖拽 → 调试预览 → diff 高亮"，但没说运算到底在哪一侧执行。架构（03 §3.4）只描述了后端 `Engine.Run` 接口。

**裁决**：**v1 全部在后端执行，前端只展示**。理由：
- PRD §6.1 性能指标 < 500ms（100 节点 + 6 算子）在后端原生 Go 是 trivial；浏览器 JS 端要重新实现一遍 6 个算子算法是浪费工程。
- 调试预览的 diff 是结果 + 输入做 set diff，由后端一次性返回 `RunPipelineResponse.steps[]`（契约 §M-PIPE.41-49 + DTO 2.9-2.10 已对齐）。
- 前端只需要：(a) 拖拽生成 AST → `POST /api/pipelines/ast-to-yaml` 拿 YAML；(b) 修改 YAML → `POST /api/pipelines/yaml-to-ast` 拿 AST；(c) 点"运行预览" → `POST /api/pipelines/:id/run { debug: true }` 拿 steps。

**需修订**：
- **需修订 docs/02-ui-design.md "组件 6 PipelineCanvas"**：明确写"所有运行/预览均由后端执行；前端不持有算子代码"。标记交后续 Agent 处理。

### 1.8 agent ↔ hub 协议版本协商（架构 §10 / ADR 0003 / PRD §6.5）

**冲突**：编码规范 §10.2 要求"hub 兼容 N-1 版本 agent；agent 启动时上报版本号；不兼容返 `AGENT_VERSION_UNSUPPORTED`"，但契约 §3 错误码表里没有 `ERR_AGENT_VERSION_UNSUPPORTED`，wsproto 协议中也没有版本协商失败时的 bye reason 标识。

**裁决**：
- 在 `pkg/agentlib/protocol.go` 已有 `const ProtocolVersion = "1.0"`，hub 在收到 `Hello` 时拿 `HelloPayload.Version` 与自身 `ProtocolVersion` 比对 major 版本号。
- major 不一致：发送 `bye { reason: "version_unsupported" }` 后关闭连接，agent 端日志告警。
- 加错误码 `ERR_AGENT_VERSION_UNSUPPORTED` HTTP 426（Upgrade Required）。
- 加常量 `BYE_REASON_VERSION_UNSUPPORTED = "version_unsupported"`。
- agent 端在 `cmd/agent/internal/transport/client.go` 处理该 bye reason 时"停止重连"（避免无限重连），并日志 + console 打印"need upgrade"。

**需修订**：
- **需修订 docs/04-api-contract.md §3 错误码表**：新增 `ERR_AGENT_VERSION_UNSUPPORTED`（426）。
- **需修订 internal/types/api.go**：补常量 `ErrAgentVersionUnsupported`。
- **需修订 pkg/agentlib/protocol.go**：新增 bye reason 常量。
- 标记交后续 Agent 处理；任务 T-14（agent hub）落地。

## 2. 遗漏补充

### 2.1 错误处理与边界情况

PRD / UI / 架构均未系统性描述以下边界，本节统一裁决：

| 场景 | 处理 | 落地任务 |
|------|------|----------|
| 用户上传 yaml 文件超过 10MB | 后端拒绝 + 返回 `ERR_VALIDATION_OUT_OF_RANGE` + UI toast 提示文件过大 | T-12 |
| 订阅 URL DNS 解析失败 / TLS 握手失败 / 5xx | 写 `last_sync_status=error` + `last_sync_error`；触发 M-NOTIFY `subscription_sync_failed`；UI 列表显示红色徽章 | T-12 |
| 节点 raw_uri 字段超过 4096 字符 | 截断到 4096 + UI 警告"原 URI 过长被截断" | T-13 |
| 流水线 100 节点 + 算子链路超过 5s 超时 | hub 强制 cancel + 返回 `ERR_PIPELINE_RUN_TIMEOUT`；UI 显示部分结果 + 错误提示 | T-22 |
| 算子 regex_rename 表达式编译失败 | 不 panic，返回 `ERR_VALIDATION_REGEX_COMPILE` + 字段 details；UI 表单字段红色标错 | T-22 |
| TCPing 节点数 > 200 | 服务端拒绝 + `ERR_VALIDATION_OUT_OF_RANGE`；UI 强制分批 | T-19 |
| Telegram Bot webhook 收到非白名单 chat_id | 静默丢弃 + 日志记录 | T-27 |
| agent 在 hello 后 90s 内无心跳 | hub 标记 status=offline + 触发 SSE `agent_status` 事件 + 触发 M-NOTIFY `node_offline` | T-14 |
| OTA 下载校验失败（SHA-256 不匹配） | 拒绝 apply + 写审计 + 触发 M-NOTIFY；不重启进程 | T-29 |
| 数据库 SQLITE_BUSY 触发 | tx wrapper 内部最多重试 3 次（exponential backoff 50/100/200ms）；仍失败返 `ERR_INTERNAL_DATABASE` | T-3 |
| 用户 session 在请求中过期 | middleware 返 401 `ERR_AUTH_TOKEN_EXPIRED`；前端 axios interceptor 自动跳 `/_app/<前缀>/login` | T-5 / T-7 |
| 静默模式前缀错误 | middleware 直接返 404 + nginx 默认页 + `Server: nginx` header | T-3 |

### 2.2 加载态 / 空状态 / 错误态（每个数据驱动组件强制三态）

每个数据驱动组件（DataTable、TrafficChart、StatusBadge 等）必须实现三态。**任何数据驱动组件 PR 缺三态实现即拒绝合入**。

- **加载态**：用 `components/ui/skeleton.tsx`（任务 0 提供）。表格：行替换为 skeleton + 顶部 1px primary progress；卡片：内容替换为 skeleton；图表：图区替换为 skeleton placeholder。
- **空状态**：中央 illustration（lucide-react 的 inbox / package / search 等图标）+ 主标题 + 副标题 + 主 CTA 按钮（如"新建订阅"）。每个空状态文案走 i18n key `<namespace>:empty.<scenario>`。
- **错误态**：banner（`components/ui/alert.tsx`）红色顶部 + 错误 message（i18n key 渲染）+ "重试"按钮。banner 5 秒后可手动关闭。
- **partial 错误**：如 200 节点 TCPing 中部分失败，列表展示部分成功结果 + 顶部 banner "部分节点测试失败 (5/200)"。

### 2.3 国际化资源加载策略

**裁决**：**懒加载 namespace**，首屏只加载 `common` + `errors` + `auth` 三个 namespace；其他 namespace 在用户进入对应路由时 lazy import。

- 首屏 JS bundle 中 i18n JSON ≤ 30KB（4 套 locale × 3 namespace）。
- 路由级 namespace 用 `i18next.loadNamespaces(['subscription'])` 在 TanStack Router `beforeLoad` 中触发。
- 翻译加载失败兜底：使用 key 本身（`t('key', { defaultValue: 'key' })`），UI 不破坏。
- 静默模式登录页（`/_public/*`）只用 `common` + `auth` namespace。
- 任务 T-2 落地懒加载机制。

### 2.4 前端登录态恢复（刷新后行为）

PRD / UI 未明确，本节裁决：

- access_token 存 `localStorage`（key `shiguang.access_token`）+ 用户 ID 存 cookie（用于 SSR-friendly 但 v1 是 SPA，cookie 主要给 SSE 用）。
- 页面刷新时：`auth-store` 初始化阶段从 localStorage 读 token → 调用 `GET /api/me` 验证 token；
  - 成功：填充 user，进入目标路由（TanStack Router 的 `beforeLoad` 守卫等待此 promise）；
  - 失败（401 ERR_AUTH_TOKEN_EXPIRED）：清 token、跳 `/_app/<前缀>/login?next=<当前 URL>`；
  - 失败（网络错误）：进入"离线"占位页 + 重试按钮（避免直接强制登出）。
- Cookie 过期与 token 过期不绑定；SSE 通过 EventSource + cookie 透传 token。
- 任务 T-7（lib/api-client）+ T-8（auth-store）+ T-6（路由守卫）落地。

### 2.5 后端启动顺序（cmd/server/main.go）

**严格按以下顺序**：

1. 解析 flag + env（`internal/config.Load`）。
2. 初始化 logger（slog json + 文件轮转）。
3. `storage.Open` 打开 SQLite（WAL + busy_timeout）。
4. `storage.Migrate`（执行 `migrations/0001_initial.sql` + `ensureColumn` 增量）。
5. 加载 `system_settings`（含静默模式前缀；首次启动时生成 32hex + 写 file + 打印到日志）。
6. 初始化 `auth.Manager`、`notify.Manager`、`agent.Hub`、`traffic.Aggregator`、`ota.Service`、`shortlink.Service`。
7. 启动后台 worker：`traffic.Aggregator` 每天 00:00；`agent.Hub.GC`（清离线 agent）每 30s；`notify.Manager.RetryWorker` 每分钟扫 pending；`session.GC` 每 10min。
8. 装配 `handler.NewRouter(deps)` HTTP mux + 中间件链。
9. 启动 `http.Server`（含 graceful shutdown）。
10. 监听 SIGTERM / SIGINT；收到信号后走退出顺序。

### 2.6 后端退出顺序（graceful shutdown）

**严格按以下顺序**：

1. 收到 SIGTERM/SIGINT → 调用 `http.Server.Shutdown(ctx, 30s)`（先停接收新请求，等正在处理的完成）。
2. `agent.Hub.Close()`（广播 `bye { reason: "server_shutdown" }` 给所有 agent，5s 等待）。
3. `notify.Manager.Close()`（flush 未发的事件，等待 timeout）。
4. `traffic.Aggregator.Close()`（停 cron）。
5. `storage.DB.Exec("PRAGMA wal_checkpoint(TRUNCATE)")` 把 WAL 合并到主库（OTA 也走此步）。
6. `storage.DB.Close()`。
7. logger flush + 退出 0。

### 2.7 单元测试与集成测试目录约定

- **Go 单测**：`*_test.go` 与被测文件同包同目录（标准做法）。
- **Go 集成测试**（跨包 / 启 httptest 服务）：`internal/handler/*_integration_test.go`，build tag `//go:build integration`，CI 在 `test.yml` 单独跑。
- **TS 单测**：`*.test.ts(x)` 与被测文件同目录。
- **E2E**：`web/e2e/<scenario>.spec.ts`，playwright 配置在 `web/playwright.config.ts`。
- **fixture**：`internal/<module>/testdata/`（Go 标准约定）+ `web/src/__fixtures__/`（前端）。

### 2.8 审计日志策略（编码规范 §6 补充）

每个写操作 handler 必须在 audit middleware 中记录：

- `action`：`login` / `logout` / `create_subscription` / `delete_subscription` / `sync_subscription` / `create_pipeline` / `delete_user` / `ota_apply` / `silent_mode_rotate` / `rotate_share_token` / `rotate_agent_token` / `enable_2fa` / `disable_2fa` 共 14 类。
- 失败的写操作（如 401 / 403 / 500）也记 audit，`success=false`。
- 读操作不记 audit（避免日志爆炸）；除了 admin 的 `GET /api/admin/audit` 本身记一条"读审计"。

### 2.9 一键安装脚本（PRD M-AGENT-1 / 流程 4）

PRD 与架构均提到 `install-agent.sh` 但没规范输入 / 输出：

- 脚本签名：`curl -fsSL https://<hub-host>/_app/<前缀>/install-agent.sh | TOKEN=xxx AGENT_NAME=vps-hk-01 bash`。
- 脚本逻辑：检测 OS/arch → 下载对应二进制（`/_app/<前缀>/dl/agent-linux-amd64`）→ 校验 SHA-256（hub 返回的下载 URL 同时给出校验和）→ 写 systemd unit → 启动。
- hub 端新增端点（不计入 102，归 M-OPS）：
  - `GET /_app/<32hex>/install-agent.sh`（admin 在 agent 创建后从 UI 复制 token 并粘贴此 URL；后端返回填好 TOKEN 的脚本）。
  - `GET /_app/<32hex>/dl/agent-<os>-<arch>` 静默模式下提供 agent 二进制下载（embed.FS）。
- 任务 T-14（M-AGENT）落地。

### 2.10 后端 i18n（仅错误码 + 邮件模板）

架构 §3.12 描述了后端 `internal/i18n`，但具体使用场景未明：

- 后端不渲染错误消息文本，只返回 `code` 字符串（编码规范 §8.4）；前端 i18n 渲染。
- **后端 i18n 仅用于邮件**：SMTP 渠道发送的邮件正文需要按收件人 locale 渲染（用户配置 `users.locale`），故后端需要 4 套 locale 邮件模板（`internal/i18n/email_templates/<locale>/<event>.tmpl.html` + `.tmpl.txt`）。
- 任务 T-27（M-NOTIFY）落地。

## 4. 技术风险

按"风险描述 / 触发概率 / 影响 / 缓解措施"四要素列出。

### 风险 1：流水线 100 节点性能预算（500ms）紧张

- **描述**：100 节点 + 6 算子 + 调试预览（每算子返回 diff）的总执行时间要 < 500ms。filter 表达式解析、regex_rename 重编译、JSON 序列化都是潜在开销。
- **触发概率**：**中**。如果初次实现 filter 用 reflect 大量调用，会接近上限。
- **影响**：**中**。功能依旧能跑，只是验收 PRD M-PIPE.3 标准不达标。
- **缓解措施**：
  - 任务 T-19 内置性能基线测试（go test -bench），CI 跑；
  - filter 表达式 evaluator 预编译 + cache（同 PipelineID 复用）；
  - regex 用 sync.Pool 缓存 *regexp.Regexp；
  - debug=false 时不收集 diff，节省 50% 时间。

### 风险 2：goja 脚本超时 kill goroutine 泄漏

- **描述**：goja Interrupt 是协作式中断（依赖脚本主动 yield），死循环 + 同步代码无法立即中断。极端情况下 goroutine 不退出 → 内存泄漏。
- **触发概率**：**中**。恶意脚本 / 用户失误（如 `while(true){}`）。
- **影响**：**重**。长期运行后内存爆。
- **缓解措施**：
  - 严格在每次调用前用 `vm.SetInterruptHandler` 限制 op count（如 1e7 ops），goja 文档推荐做法；
  - 配置 `runtime.SetGCPercent` 监控；
  - 任务 T-13 单测覆盖死循环 case；
  - 失败 3 次的脚本自动 disable + 通知用户。

### 风险 3：WAL 模式下备份 / OTA 时的 SQLite 一致性

- **描述**：直接复制 .db 文件可能丢失 WAL 中尚未 checkpoint 的数据；OTA 自更新时若 wal_checkpoint 未执行，重启后可能出现部分丢失。
- **触发概率**：**低**。需要 WAL 中有 pending 写 + 异常重启。
- **影响**：**重**。数据不一致 + 用户信任损失。
- **缓解措施**：
  - 任务 T-26 备份流程强制先 `PRAGMA wal_checkpoint(TRUNCATE)` 再复制；
  - 任务 T-27 OTA applier 同上 + 用 `.bak` 暂存原文件，回滚机制；
  - 退出顺序（§2.6）严格执行；
  - 集成测试 T-34 覆盖"备份 → 模拟崩溃 → 恢复"完整路径。

### 风险 4：静默模式前缀泄露 + 用户体验

- **描述**：用户在群里截图 / 复制完整 URL → 32hex 前缀泄露 → 攻击者直接命中后台。普通用户没意识 risk。
- **触发概率**：**中**。
- **影响**：**中**。攻击者仍要过 token + 2FA，但暴露了项目特征。
- **缓解措施**：
  - 任务 T-26 在 UI 提供"复制 URL"按钮的 toast 警告"切勿截图分享"；
  - 提供轮换按钮（手动）；
  - 文档（T-33）强调风险；
  - 未来 v1.1 加"信任设备 cookie + 无前缀"逃生方案。

### 风险 5：Nezha v2 协议字段映射不全 / 上游演进

- **描述**：Nezha 上游持续演进，v2 字段在不同 minor 版本可能加新字段；如果只兼容最小子集，上游加字段后 hub 会丢弃额外信息但 agent 仍发送（白噪声）。
- **触发概率**：**高**。Nezha 团队活跃。
- **影响**：**低**。字段被忽略不影响 hub 功能。
- **缓解措施**：
  - 任务 T-17 在解析时用 `json.Unmarshal` 到 struct + `map[string]json.RawMessage` 兜底；未知字段日志 warn；
  - 季度评审 Nezha 协议变更；
  - 后续如有用户提需求扩展，按 ADR 0003 附录 A 补；
  - 不在 9 周路线图内做完整 v1/v2 兼容（PRD §8 Flagged 已定）。

### 风险 6：前端工作量评估偏低

- **描述**：22 页面 + 15 组件 + 4 套 locale × 11 namespace = 大量文案；UI 文档 1112 行，复杂度高（特别 PipelineCanvas + Cmd+K + DataTable 通用性）。
- **触发概率**：**中-高**。
- **影响**：**中**。延期 1-2 周。
- **缓解措施**：
  - 任务拆分中前端任务工时（T-2 4 人日、T-20 + T-21 共 4 人日）已留余量；
  - 共享组件（T-2）一次性投入，后续业务页面成本 0.5-1.5 人日；
  - 4 套 locale 中只有 zh-CN 是手写，其他 3 套用 LLM 翻译 + 人工校对；
  - Buffer 在 W11 留出 1 周；
  - 实在不行可砍 P1 功能（如 audit 导出 / 备份 S3）。

### 风险 7：依赖 modernc.org/sqlite 在某些平台的 bug

- **描述**：modernc 是纯 Go 移植，性能与稳定性比 mattn/go-sqlite3 略差；曾在 arm64 macOS 出现过 bug。
- **触发概率**：**低**。
- **影响**：**重**。如果在某平台无法运行，跨平台承诺破灭。
- **缓解措施**：
  - 任务 T-32 CI 矩阵在 5 个平台都跑 `go test` + smoke test；
  - 早期（W3）就在 linux/amd64 + darwin/arm64 + windows/amd64 三平台跑 T-1 单测；
  - 若发现致命 bug，降级到 mattn/go-sqlite3（牺牲交叉编译，回退到 cgo 路径），并修 ADR 0001。

### 风险 8：WebSocket 在某些 NAT / CDN 场景下被截断

- **描述**：Cloudflare 免费版 / 某些反代默认会断长连接（默认 100s 超时）。agent 频繁断线 → presence 抖动 → 误触发 node_offline 通知。
- **触发概率**：**中**。Cloudflare 用户常见。
- **影响**：**中**。可用性受影响。
- **缓解措施**：
  - 任务 T-14 心跳 30s + presence 90s 容忍 2 次未到（90s 超时不立即标 offline，再过 30s 才标）；
  - 任务 T-15 agent 端断线后立即重连（指数退避，初始 1s）；
  - 文档（T-33）建议用户用 Cloudflare 时启用 "WebSocket support" + 配合反代 keepalive；
  - v1.x 路线图加 HTTPS long-polling fallback（PRD §7.1 P2）。

### 风险 9：Telegram Bot webhook 在国内不可达

- **描述**：国内 VPS 无法访问 api.telegram.org，但用户的"hub 部署在国内 + 通知发到 TG"是常见场景。
- **触发概率**：**高**（中国用户）。
- **影响**：**中**。可降级到长轮询，但实现复杂度上升。
- **缓解措施**：
  - 任务 T-24 默认走 webhook（境外 VPS / 海外用户）；
  - 提供"代理服务器"配置项（HTTP_PROXY 环境变量）；
  - 国内用户可优先用 Bark / Server酱 / PushDeer（这三家是国内独家，已在 10 渠道清单）；
  - 文档（T-33）明确指引。

### 风险 10：4 套 locale 的 key 一致性维护成本

- **描述**：CI 校验 4 套 locale 的 key 集合必须相等；新增 key 必须 4 套同步加。如果某次 PR 漏了，CI fail 阻塞合入。
- **触发概率**：**高**。常见。
- **影响**：**低**。CI 拦截，但拖慢开发节奏。
- **缓解措施**：
  - 任务 T-30 `check-i18n.sh` 第一次 fail 时友好提示"缺失的 key 列表"；
  - PR 模板加 "i18n 4 套已同步" 勾选项；
  - 写一个 `pnpm i18n:sync` 辅助脚本，从 zh-CN 拷贝缺失 key 到其他 3 套（值为 [TODO] 占位），开发者再翻译。

## 5. 简单化检查（去过度设计）

### 已识别的可简化项

1. **流水线编辑器的"YAML 视图 mirror"**：UI §流程 3 描述左侧 Monaco + 右侧 GUI mirror 只读视图。本计划裁决：**砍掉右侧 mirror**，YAML 视图只展示 Monaco 全宽。理由：双向同步走 debounce 300ms，用户改 YAML 后切回 GUI tab 就能看到效果；mirror 是冗余。**需修订 docs/02-ui-design.md "组件 6 PipelineCanvas / YAML 视图"**。
2. **`internal/types/agentproto/` 包**：架构 §2 列出该子包，但实际 `pkg/agentlib/protocol.go` 已经承担同样职责。**裁决：删除 `internal/types/agentproto/` 子包计划**，所有 ws proto 共享走 `pkg/agentlib/` + `internal/types/wsproto.go`。
3. **`internal/handler/middleware/cors.go`**：当前默认禁用（架构 §3.1）。**裁决：v1 不实现 cors middleware**（前端与后端同源部署，不需要）；P2 真有需要时再加。
4. **多个 admin 端点合并**：契约里 `/api/admin/agents` 与 `/api/agents`（参数 ?all=true）功能重叠。**裁决：保留两条路径但共用同一 handler 函数**（按 user role 决定查询范围）。
5. **DataTable 高度参数化**：UI §组件 3 列出 DataTable 极多参数（列设置 / 导出 / 批量 / 视图切换 / 响应式卡片）。**裁决：v1 实现 6 个核心特性（搜索 / 排序 / 分页 / 选中 / 加载态 / 空态）；列设置 + 导出 + 响应式卡片放 P1（v1.1）**。

### 不能砍的（即使诱人）

- 静默模式：是核心差异化（PRD §M-OPS）。
- 4 套 locale：是产品定位，砍了等于改产品。
- 算子流水线 GUI：是差异化 #1。
- 10 渠道通知：是差异化 #2，国内渠道（Bark / Server酱 / PushDeer）尤其不能砍。
- TOTP 2FA：是安全红线（PRD §6.3）。
