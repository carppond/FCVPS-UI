# ADR 0002: 用户模型 —— 多用户 + admin/user 二分权限

## Context

- 妙妙屋当前是**单用户**模型，1 个面板 1 个登录账号，导致"小团队共享一份订阅 + 探针"场景必须共用账号，安全审计困难。
- 调研对比（详见 docs/_research-competitors.md 第二节）：
  - Nezha / Beszel / 3X-UI / Xboard 有多用户能力，是同类垂直领域常见诉求。
  - Xboard 走完整 RBAC（角色 + 权限 + 资源域），但其定位是商业 SaaS。
- 本项目目标用户为：个人技术爱好者 / 小机场主 / 5-10 人技术小团队。完整 RBAC 在该规模下的边际收益低，且增加 schema 与 UI 复杂度。

## Decision

采用 **多用户 + 二分权限（admin / user）** 模型：

- `admin`：拥有所有功能（用户管理、系统设置、OTA 升级、查看所有用户数据等）。
- `user`：可管理自己的订阅、节点、规则、脚本、通知、短链、API token，但**无法访问其他用户的数据**，也无法修改系统设置。
- 首次部署的初始账号默认为 admin。
- 不引入"自定义角色 / 资源级 ACL"等完整 RBAC 能力。

## Consequences

**正面**：
- 数据模型简单：用户表加一个 `role` 字段（enum: admin / user），所有业务表加 `user_id` 外键即可。
- 权限检查代码简短：中间件层 + 业务层各一道；admin 跳过 user_id 过滤，user 强制 user_id = self。
- 5-10 人小团队场景下覆盖率足够，与目标用户匹配。

**负面 / 待办**：
- 不支持"团队 / 部门"层级（小团队当下无诉求；如未来需要走 v2 路线再扩 RBAC）。
- 不支持"细粒度操作权限"（如 user 只读、user 仅订阅不可见探针等）。在 v1 这是 by design 的简化。
- 资源转让能力（如 admin 把某个订阅转给 user）暂未设计，已在 CONTEXT.md Flagged 标注。

**替代方案为何放弃**：
- **完整 RBAC（Casbin / Xboard 风格）**：复杂度高，UI 端需要"角色编辑器"，对个人 / 小团队场景过度设计。
- **保持单用户（妙妙屋现状）**：阻断"小团队共享面板"用例，差异化竞争力下降。
- **OAuth/OIDC 接入企业 SSO**：目标用户群体绝大多数无企业 SSO 设施，需求伪刚需；列入 P2 后期。
