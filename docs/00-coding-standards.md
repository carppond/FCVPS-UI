# 拾光VPS 编码规范

> 本文档是 shiguang-vps 项目的硬约束，适用于后端（Go）、前端（React/TS）、Agent（Go）三端。
> 所有 PR 必须通过本规范的 checklist 才能合入。新人入项目第一份要读的文档。

---

## 1. 文件粒度

防止"上帝文件 / 上帝函数"。参考反面教材：miaomiaowu 项目存在 162KB 的单文件 Go 源码，调试与代码审查成本极高，本项目严禁出现。

| 维度 | 上限 | 说明 |
| --- | --- | --- |
| Go 单文件 | ≤ 500 行 | 超过必须按职责拆分（handler / repo / service 分离） |
| Go 单函数 | ≤ 80 行 | 含空行与注释。超过抽取为子函数 |
| TS / TSX 单文件 | ≤ 300 行 | 超过拆 hook / 子组件 / utils |
| TS 单函数 | ≤ 50 行 | 包括 React 组件函数体 |
| React 单组件 | ≤ 200 行 | 视图与逻辑分离，逻辑沉到 hooks |
| Agent 进程总代码量 | ≤ 5000 行 | 保持轻量。重逻辑放 hub，不放 agent |

### 1.1 例外
- 自动生成的代码（如 `types/` 中由 Go 生成的 TS 类型）不受文件行数限制，但必须在文件头注明 `// Code generated by xxx; DO NOT EDIT.`
- locale JSON 不计入行数限制。

## 2. 注释要求

### 2.1 Go
- 公共（导出）函数、类型、接口、常量必须有 godoc 注释，首字母大写开头，完整句号结尾，名字与注释开头一致：

  ```go
  // ParseClashYAML parses a Clash subscription document into an ordered YAML node tree.
  // It preserves field order and quote style so the original formatting is not lost.
  func ParseClashYAML(raw []byte) (*yaml.Node, error) { ... }
  ```

- 内部（非导出）函数仅在"为什么这样做"不明显时写注释。已经一眼能看懂 what 的不必加注释。
- 行内注释解释 why，不解释 what：

  ```go
  // OK
  // SQLite WAL mode requires writer to hold the file lock; do not call from a read-only tx.
  if err := tx.Exec(...); err != nil { ... }

  // BAD
  // execute the sql
  if err := tx.Exec(...); err != nil { ... }
  ```

### 2.2 TS / TSX
- 公开导出的 hook / util / 复杂组件 props 必须有 JSDoc。
- 简单的 UI 组件可省略，但 props 类型必须明确（不允许 `any`）。

### 2.3 通用
- TODO 必须带原因和（可选）日期：

  ```ts
  // TODO(perf): 该列表 N 大于 1000 后必须接入 react-virtual
  // TODO(2026-Q3): 后端 sub-store 兼容层弃用后移除
  ```

- 严禁注释掉的代码留在仓库里。需要保留的旧实现请走 git 历史或写在 PR 描述里。
- 注释里禁止出现密码、token、内部 URL 等敏感信息。

## 3. 代码复用

### 3.1 三次原则
同样的逻辑出现 **3 次** 必须提取为共享函数 / 组件。第二次可以容忍 copy，第三次必须重构。

### 3.2 目录归属

**Go**
- `internal/util/`：纯函数工具（无副作用、无外部依赖）。如 `util/yaml.go`、`util/strslice.go`。
- `internal/types/`：跨包共享的结构体、接口、错误码常量。
- `pkg/`：仅当确实需要被外部项目 import 时才用，否则一律放 `internal/`。

**TS**
- `src/lib/`：纯工具函数（`cn`、`storage`、`format`、`theme` 等）。
- `src/types/`：从后端同步的 API 类型 + 前端独有的视图类型。
- `src/components/ui/`：与业务无关的原子组件（封装 Radix + Tailwind 之后的 `Button` / `Dialog` / `Input` / `Select`）。
- `src/hooks/`：通用 hook（`useDebounce` / `useMediaQuery` / `useLocalStorage`）。业务 hook 放在对应业务目录下。

### 3.3 API 契约类型
- 单一事实源在 Go：`internal/types/api/`。
- 通过工具（推荐 [tygo](https://github.com/gzuidhof/tygo)）从 Go struct 生成 TS 类型，落到 `src/types/api/`。
- 禁止前后端各自维护一份类型，禁止手工同步。
- 生成脚本写入 `scripts/gen-types.sh`，CI 强制校验"生成后无 diff"。

## 4. 命名规范

### 4.1 Go
| 对象 | 规范 | 示例 |
| --- | --- | --- |
| 包名 | 全小写单词，无下划线 | `pipeline`、`scriptengine`、`nezha` |
| 文件名 | snake_case | `yaml_helpers.go`、`auth_middleware.go` |
| 类型 | PascalCase | `Subscription`、`PipelineRunner` |
| 接口 | PascalCase，倾向 -er 后缀 | `Storer`、`Notifier` |
| 导出函数 / 常量 | PascalCase | `ParseClashYAML`、`DefaultTimeout` |
| 内部函数 / 变量 | camelCase | `parseLine`、`maxRetries` |
| 错误变量 | Err 前缀 | `ErrUserNotFound`、`ErrInvalidToken` |

### 4.2 TS / TSX
| 对象 | 规范 | 示例 |
| --- | --- | --- |
| 文件名 | kebab-case | `user-list.tsx`、`use-debounce.ts` |
| React 组件（导出名） | PascalCase | `UserList`、`PipelineEditor` |
| 变量 / 普通函数 | camelCase | `currentUser`、`buildPayload` |
| 类型 / 接口 | PascalCase | `Subscription`、`PipelineNode` |
| 常量（顶层 immutable） | SCREAMING_SNAKE_CASE | `MAX_RETRY`、`DEFAULT_LOCALE` |
| React hook | use 前缀 | `useSubscription` |

### 4.3 布尔变量
必须以 `is` / `has` / `should` / `can` 开头：

```ts
// OK
const isLoading = ...
const hasError = ...
const canEdit = ...

// BAD
const loading = ...
const error = ... // 这是 string/Error 还是 boolean？
```

### 4.4 黑名单
禁止以下无意义命名出现在作用域 ≥ 5 行的位置：

`data`、`info`、`temp`、`result`、`val`、`obj`、`item`（在循环作用域 < 5 行内可用）、`foo` / `bar`。

例外：常见迭代变量 `i` / `j` / `k` 可用，但仅限简单循环。

## 5. 性能要求

### 5.1 通用
- 循环嵌套 ≤ 3 层。第 4 层必须重构（用 map 索引、提前 break、抽函数）。
- 无依赖的异步操作必须并行：
  - Go：`golang.org/x/sync/errgroup` 或 `goroutine + chan`
  - TS：`Promise.all` / `Promise.allSettled`

```go
// OK
g, ctx := errgroup.WithContext(ctx)
for _, sub := range subs {
    sub := sub
    g.Go(func() error { return fetchOne(ctx, sub) })
}
if err := g.Wait(); err != nil { ... }
```

```ts
// OK
const [profile, subs, agents] = await Promise.all([
  api.getProfile(),
  api.listSubscriptions(),
  api.listAgents(),
]);
```

### 5.2 前端
- 列表 render 必须有稳定 key（不能用 index 作为 key，除非列表完全静态）。
- ≥ 100 行的可滚动列表必须接入 `@tanstack/react-virtual`。
- 路由级代码分割：所有 `src/routes/` 下的页面必须用 `lazy` + `Suspense` 加载。
- 大组件（含复杂 props 比较）用 `React.memo`，配合 `useMemo` / `useCallback` 稳定引用。
- 严禁在 render 函数中 new 对象 / 数组当 prop 传下去（每次新引用会击穿 memo）。

### 5.3 后端
- 禁止 N+1 查询：批量场景必须用 `IN (...)` / join / 一次查询 + 内存聚合。
- 高频查询字段必须加索引（在 migration 文件中显式写 `CREATE INDEX`）。
- SQLite 必须启用 WAL 模式（`PRAGMA journal_mode=WAL`），连接池写连接数 = 1。
- p99 ≤ 10ms 的 hot path 不走 DB：用 `sync.Map` 或 LRU（如 `hashicorp/golang-lru`）做内存缓存，附 TTL。

## 6. 错误处理

### 6.1 Go
- 所有外部调用（DB / HTTP / 文件 / 子进程）都必须 `if err != nil` 处理，禁止 `_`。
- 错误包装使用 `fmt.Errorf("op: %w", err)`，保留错误链：

  ```go
  if err := repo.Save(ctx, sub); err != nil {
      return fmt.Errorf("save subscription %s: %w", sub.ID, err)
  }
  ```

- 业务错误用 sentinel error 定义在所属包顶层，方便 `errors.Is` 判断：

  ```go
  var (
      ErrUserNotFound   = errors.New("user not found")
      ErrInvalidToken   = errors.New("invalid token")
      ErrSubExpired     = errors.New("subscription expired")
  )
  ```

- HTTP handler 必须有 panic 兜底中间件（`recover()` + log + 返回 500 错误码），单 handler 崩溃绝不能让进程退出。
- 不要返回 `error` 后又自己 log 一次（要么 wrap return，要么终结点 log，避免重复日志）。

### 6.2 TS
- 所有异步操作必须 `try/catch` 或 `.catch()`。Promise chain 末端必须有 catch。
- 错误信息走 i18n：

  ```ts
  // OK
  toast.error(t('errors.subscription.fetchFailed'));

  // BAD
  toast.error('Failed to fetch subscription: ' + err.message);
  ```

- 严禁空捕获：

  ```ts
  // BAD
  try { ... } catch (e) {}

  // OK
  try { ... } catch (e) {
    logger.warn('non-critical failure', e);
  }
  ```

- 严禁吞掉非 Error 类型，统一用 `if (e instanceof Error)` 判断后处理。

### 6.3 通用
- 用户可见错误必须友好提示，不能暴露内部路径 / SQL / 堆栈。
- 错误日志必须带上下文：用户 ID（脱敏）、请求 ID（trace_id）、时间戳、操作名。
- 错误码与文案分离：后端只返回 code（字符串枚举），message 由前端 i18n 渲染。详见 §8。

## 7. 项目结构

### 7.1 后端（Go）

```
cmd/
  server/              # 主程序入口 (hub)
    main.go
  agent/               # agent 入口（独立单二进制）
    main.go
internal/
  auth/                # 登录、2FA、token、session
  handler/             # HTTP handler，按业务域切分子包
    handler_sub.go
    handler_agent.go
    handler_pipeline.go
    middleware/        # recover / cors / authn / authz
  storage/             # SQLite repo 层，唯一直接接触 DB 的层
    sqlite/
    repo_sub.go
    repo_user.go
  substore/            # sub-store 兼容 / 多格式 producer（clash/sing-box/v2ray/...）
  pipeline/            # ★ 算子流水线引擎
    operator/          # 内置算子
    runner.go
  notify/              # ★ 通知通道插件（telegram / bark / webhook / email）
    channel/
  scriptengine/        # goja 沙箱 + 资源限额
  agent/               # agent 客户端业务逻辑（与 cmd/agent 共享）
  nezha/                # Nezha 协议兼容层（独立包，不污染自有协议）
  logger/              # zap / slog 封装
  util/                # 纯函数工具
    yaml.go            # YAML Node 操作 helper
    strslice.go
  types/               # 跨包共享类型 + API 契约
    api/
pkg/                   # 仅当真要被外部 import 时使用
migrations/            # SQL migration（与 ensureColumn 模式二选一）
```

### 7.2 前端（React）

```
src/
  routes/                  # TanStack Router 路由文件
    __root.tsx
    _authed/
      subscriptions.tsx
      pipeline.$id.tsx
  components/
    ui/                    # 原子组件（Radix + Tailwind 封装）
      button.tsx
      dialog.tsx
    pipeline/              # 算子流水线相关业务组件
    notify/                # 通知配置 UI
    agent/                 # agent 监控 UI
    layout/                # AppShell / Sidebar / TopBar
  hooks/                   # 通用 hook
  lib/                     # cn / storage / theme / format
  stores/                  # Zustand stores（auth / ui-preference）
  api/                     # API 客户端（基于契约类型，含 query keys）
  locales/                 # i18n 资源
    zh-CN/
    en/
    ja/
    ko/
  types/
    api/                   # 从后端 Go 同步生成的 TS 类型（不要手改）
  styles/                  # 全局样式 / Tailwind 入口
```

### 7.3 Agent（Go）

```
cmd/agent/main.go
internal/agent/
  collector/               # 指标采集：CPU / MEM / 带宽 / 进程
  transport/               # WebSocket 客户端（重连 / 心跳 / 协议编解码）
  nezha_compat/            # Nezha 兼容输出，可单独开关
```

### 7.4 跨层调用规则（硬约束）

- `handler/` 不能直接接触 SQLite，必须通过 `storage/`。违反则编译期报错（用 `internal/` 包可见性约束 + `go vet` 自定义规则）。
- `storage/` 不能 import `handler/`、`pipeline/`、`notify/`（防止环依赖）。
- `components/ui/` 不能 import 任何业务 `stores/` 或 `api/`，必须保持纯展示。
- `pipeline/operator/` 内的算子之间不互相 import；公共能力沉到 `pipeline/` 顶层或 `util/`。
- agent 二进制不能 import hub 才有的包（如 `internal/handler/`、`internal/storage/sqlite/`）。

## 8. i18n 硬约束

### 8.1 字符串
- 所有用户可见字符串（按钮文案、表单 label、toast、tooltip、空状态、错误提示）必须走 `t('namespace.key')`。
- 严禁硬编码中文、英文、日文、韩文出现在 `.tsx` / `.ts` 业务文件里。
- 例外：开发期 `console.log`、技术性日志、纯英文标识符（如 `Authorization`、`Bearer`）。

### 8.2 命名空间组织

```
locales/
  zh-CN/
    common.json          # 通用：保存 / 取消 / 删除 / 确认 / 加载中
    subscription.json    # 订阅模块
    pipeline.json        # 流水线模块
    notify.json          # 通知模块
    agent.json           # agent 模块
    errors.json          # 错误码 → 文案
  en/ ja/ ko/             # 同结构对齐
```

- 每个命名空间在 4 套 locale 下 key 必须完全一致（CI 校验）。
- 不允许"中文最全、其他语言只翻译一半"。新增 key 必须 4 套同步加。

### 8.3 数字 / 日期 / 流量格式
- 数字格式化用 `Intl.NumberFormat(locale, ...)`，禁止 `num.toLocaleString()` 不带 locale。
- 日期格式化用 `date-fns/locale`：

  ```ts
  import { format } from 'date-fns';
  import { zhCN, enUS, ja, ko } from 'date-fns/locale';
  format(date, 'PPpp', { locale: zhCN });
  ```

- 流量单位（B / KB / MB / GB / TB）封装到 `lib/format.ts`，并对单位做 i18n（`t('common.unit.gb')`）。

### 8.4 后端错误码契约
- 后端只返回 `code`（字符串枚举，例如 `SUBSCRIPTION_EXPIRED`、`AGENT_OFFLINE`），不返回翻译后的 message。
- 前端在 `errors.json` 中按 code 维护 4 套翻译。
- 后端 code 字符串集中定义在 `internal/types/api/errcode.go`，TS 一侧自动同步。

## 9. YAML 节点操作规范

订阅文件（Clash 配置）必须保留字段顺序、引号风格、注释，否则用户编辑过的订阅会被破坏。

### 9.1 必须
- 使用 `gopkg.in/yaml.v3` 的 `yaml.Node`（保留顺序、Style、HeadComment）。
- 所有 YAML 结构改写走 `internal/util/yaml.go` 的 helpers：
  - `ReorderProxyNode(n *yaml.Node, order []string)`
  - `SetMappingValue(n *yaml.Node, key string, val *yaml.Node)`
  - `CloneNode(n *yaml.Node) *yaml.Node`
- 序列化必须使用 `yaml.Encoder` + `SetIndent(2)`。

### 9.2 禁止
- 严禁先 `yaml.Unmarshal(&map[string]any{})` 再 `yaml.Marshal`：会丢失顺序、引号、注释。
- 严禁手工拼字符串生成 YAML（缩进、转义错一个就全坏）。

### 9.3 测试
- `internal/util/yaml.go` 的每个 helper 必须有"输入 == 输出"的快照测试（在不修改语义时输入 raw bytes 与输出 raw bytes 必须一致）。

## 10. agent 与 hub 的兼容性约束

agent 部署在用户的 VPS 上，升级周期长，必须考虑长期兼容。

### 10.1 协议字段
- agent 上报协议 / hub 下发协议一旦发布到 release，字段永远向后兼容：
  - 不能删字段（标记为 deprecated 但保留接收）
  - 不能改字段类型（如 `int` 改 `string` 是 break change）
  - 不能改字段语义（如单位从 byte 改 KB 是 break change）
- 必须新增时：增加新字段，老 agent 不发即视为零值/默认。
- 协议结构体集中在 `internal/types/agentproto/`，每次改动必须在 PR 描述里附"兼容性影响"评估。

### 10.2 版本支持
- 新版 hub 必须兼容上一个大版本（N-1）的 agent；agent 升级允许滞后用户一个大版本。
- agent 启动时上报版本号，hub 在不兼容时返回明确的错误码（`AGENT_VERSION_UNSUPPORTED`）并日志记录。

### 10.3 Nezha 兼容
- Nezha 协议适配层完整收敛在 `internal/nezha/`，外部包通过适配器接口（`nezha.Adapter`）使用，不允许把 Nezha 字段渗透到自有协议结构体里。
- Nezha 兼容是"附加能力"，自有协议路径必须能在不依赖 Nezha 包的情况下编译运行。

## 11. 测试要求

### 11.1 Go
- 必须有单测的包：
  - `internal/util/`（纯函数，100% 覆盖）
  - `internal/pipeline/`（算子流水线引擎，含每个内置算子）
  - `internal/scriptengine/`（goja 沙箱：超时、内存、白名单）
  - `internal/substore/`（解析器与 producer 必须有快照测试）
  - `internal/nezha/`（兼容层每个 endpoint 一个表驱动测试）
- 覆盖率底线：`go test -cover` ≥ **50%**。不强求 80%，避免为了凑数字写无意义的 mock 测试。
- 表驱动测试为主，子测试名用 `t.Run(name, ...)` 显式标注。

### 11.2 TS
- 必须有单测的目录：
  - `src/lib/`（工具函数）
  - 算子 UI 的关键交互（拖拽、连线、参数表单 schema 校验）
- 工具：Vitest + React Testing Library。
- 覆盖率底线：核心 lib ≥ **70%**，组件不强求覆盖率。

### 11.3 E2E
- 工具：Playwright。
- 必须覆盖的关键路径：
  1. 登录 + 2FA
  2. 创建 / 编辑 / 删除订阅
  3. 创建并运行一条流水线（含至少 2 个算子）
- 在 CI 上跑 headless，每次发版前必跑。

### 11.4 测试代码规范
- 测试文件命名：Go `xxx_test.go`，TS `xxx.test.ts(x)`。
- 测试函数必须能在不依赖外部网络的情况下独立运行（用 fixture / httptest / msw mock）。
- 不在测试里用 `time.Sleep`，用 fake clock（`github.com/benbjohnson/clock` 或自封装）。

## 12. 安全 checklist（每个 PR 必查）

下列每一条都是 reviewer 在 PR 描述中必须看到的勾选项。任何一项未通过即拒绝合入。

- [ ] **用户输入是否做了正则编译错误兜底**：所有 `regexp.MustCompile` 用在用户输入上时必须改为 `regexp.Compile` + error 处理，禁止让用户构造的正则 panic 整个进程。
- [ ] **SQL 是否参数化**：所有 SQL 走 `db.QueryContext(ctx, "... WHERE id = ?", id)` 形式，禁止 `fmt.Sprintf` 拼 SQL。即便参数来源是"内部代码"也不行（防止后续重构引入注入）。
- [ ] **是否引入新的硬编码 token / secret**：如 ipinfo token、telegram bot token、第三方 API key —— 必须走环境变量或 `internal/config/`，并在 `.env.example` 中占位。`git grep` 不允许出现真实 token。
- [ ] **错误信息是否泄露内部细节**：返回给前端的错误 message 不能包含文件路径、SQL 语句、内部 IP、stack trace。这些信息只能进日志。
- [ ] **新增 endpoint 是否经过 auth middleware**：除登录、健康检查、Nezha 公开端点外，所有 HTTP / WebSocket endpoint 必须挂上 `auth.Required` 中间件。新增路由必须在 PR 中明确说明鉴权策略。
- [ ] **新增依赖是否审计**：引入新的 Go module / npm 包必须在 PR 描述中说明用途，并确认 license 与维护活跃度（避免引入弃坑或 GPL 污染）。
